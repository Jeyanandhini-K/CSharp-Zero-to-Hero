# 🎯 Interfaces in C#

## 🧠 What is an Interface?

An **interface** is a **contract** that defines what methods and properties a class must implement. It has no implementation — only signatures.

> Interface = "What must you do?" (not "How do you do it?")

---

## 📌 Basic Interface

```csharp
// Interface defines the contract
public interface IAnimal
{
    void Speak();
    void Eat();
    string Name { get; set; }
}

// Class implements the interface
public class Dog : IAnimal
{
    public string Name { get; set; }

    public void Speak()
    {
        Console.WriteLine("Woof!");
    }

    public void Eat()
    {
        Console.WriteLine("Dog is eating.");
    }
}

public class Cat : IAnimal
{
    public string Name { get; set; }

    public void Speak()
    {
        Console.WriteLine("Meow!");
    }

    public void Eat()
    {
        Console.WriteLine("Cat is eating.");
    }
}

// Usage
IAnimal dog = new Dog();
dog.Name = "Buddy";
dog.Speak();  // Woof!
dog.Eat();    // Dog is eating.
```

**Key point:** Interface name starts with `I` by convention.

## 🔗 Multiple Interface Implementation

A class can implement **multiple interfaces** (unlike inheritance).

```csharp
public interface IWalker
{
    void Walk();
}

public interface IRunner
{
    void Run();
}

public class Human : IWalker, IRunner
{
    public void Walk()
    {
        Console.WriteLine("Human is walking.");
    }

    public void Run()
    {
        Console.WriteLine("Human is running.");
    }
}

Human h = new Human();
h.Walk();  // Human is walking.
h.Run();   // Human is running.
```

## 🏆 Interface vs Abstract Class

| Feature | Interface | Abstract Class |
|---|---|---|
| Multiple inheritance | ✅ Yes | ❌ No (single) |
| Can have implementation | ❌ No | ✅ Yes |
| Constructor | ❌ No | ✅ Yes |
| Access modifiers | ❌ No (all public) | ✅ Yes (public, private, protected) |
| Purpose | Contract/behavior | Base functionality |

```csharp
// Interface - defines what you must do
public interface IPayable
{
    void ProcessPayment(double amount);
}

// Abstract class - provides base implementation
public abstract class Vehicle
{
    public abstract void Drive();
    public void Stop() { Console.WriteLine("Stopped"); }
}

// Class can inherit abstract class AND implement interface
public class Car : Vehicle, IPayable
{
    public override void Drive()
    {
        Console.WriteLine("Car is driving.");
    }

    public void ProcessPayment(double amount)
    {
        Console.WriteLine($"Payment: ${amount}");
    }
}
```

## 💡 Real-World Example: Logging System

```csharp
public interface ILogger
{
    void Log(string message);
}

public class ConsoleLogger : ILogger
{
    public void Log(string message)
    {
        Console.WriteLine($"[Console] {message}");
    }
}

public class FileLogger : ILogger
{
    public void Log(string message)
    {
        System.IO.File.AppendAllText("log.txt", $"{message}\n");
    }
}

public class EmailLogger : ILogger
{
    public void Log(string message)
    {
        Console.WriteLine($"[Email] Sending: {message}");
    }
}

public class Application
{
    private ILogger logger;

    public Application(ILogger logger)
    {
        this.logger = logger;
    }

    public void Run()
    {
        logger.Log("Application started");
        logger.Log("Processing data");
        logger.Log("Application ended");
    }
}

// Usage - can switch loggers easily
Application app1 = new Application(new ConsoleLogger());
app1.Run();

Application app2 = new Application(new FileLogger());
app2.Run();

Application app3 = new Application(new EmailLogger());
app3.Run();
```

## 🔍 Interface as a Type

Use interface reference to point to implementing objects.

```csharp
public interface IVehicle
{
    void Drive();
}

public class Car : IVehicle
{
    public void Drive() { Console.WriteLine("Car driving"); }
}

public class Bike : IVehicle
{
    public void Drive() { Console.WriteLine("Bike driving"); }
}

// Interface as type
IVehicle vehicle1 = new Car();
IVehicle vehicle2 = new Bike();

vehicle1.Drive();  // Car driving
vehicle2.Drive();  // Bike driving

// Collection of interfaces
List<IVehicle> vehicles = new List<IVehicle>
{
    new Car(),
    new Bike(),
    new Car()
};

foreach (IVehicle v in vehicles)
{
    v.Drive();
}
```

## 🔌 Dependency Injection Using Interfaces

Interfaces make code flexible and testable.

```csharp
public interface IDatabase
{
    void Save(string data);
    string Load(string key);
}

public class SQLDatabase : IDatabase
{
    public void Save(string data) { }
    public string Load(string key) { return "data"; }
}

public class MockDatabase : IDatabase  // For testing
{
    public void Save(string data) { }
    public string Load(string key) { return "mock data"; }
}

public class UserService
{
    private IDatabase database;

    public UserService(IDatabase db)
    {
        database = db;
    }

    public void RegisterUser(string name)
    {
        database.Save(name);
    }
}

// Production code
UserService service1 = new UserService(new SQLDatabase());

// Testing code
UserService service2 = new UserService(new MockDatabase());
```

## 🎤 Interview Q&A

### Q1: What is an interface?
**A:** An interface is a contract defining methods and properties that implementing classes must provide. It has no implementation.

```csharp
public interface IAnimal
{
    void Speak();
}

public class Dog : IAnimal
{
    public void Speak() { Console.WriteLine("Woof!"); }
}
```

---

### Q2: Why use interfaces?
**A:** 
- Define a contract/standard for classes
- Enable multiple implementation
- Make code flexible and maintainable
- Support dependency injection for testability

---

### Q3: Can a class implement multiple interfaces?
**A:** Yes! Unlike inheritance, a class can implement many interfaces.

```csharp
public interface IWalker { void Walk(); }
public interface IRunner { void Run(); }

public class Human : IWalker, IRunner
{
    public void Walk() { }
    public void Run() { }
}
```

---

### Q4: Can you inherit from both a class and an interface?
**A:** Yes! Class first, then interfaces.

```csharp
public class Dog : Animal, IAnimal, IFriendly
{
}
```

---

### Q5: What is the difference between an interface and an abstract class?
**A:**
- **Interface** = Contract only, no implementation, multiple allowed
- **Abstract class** = Can have implementation, single inheritance only

```csharp
// Interface
public interface IPayable { void Pay(); }

// Abstract class
public abstract class Vehicle
{
    public abstract void Drive();
    public void Stop() { }  // Can have implementation
}
```

---

### Q6: Can an interface have properties?
**A:** Yes! But only signatures (no backing fields).

```csharp
public interface IPerson
{
    string Name { get; set; }
    int Age { get; set; }
}

public class Person : IPerson
{
    public string Name { get; set; }
    public int Age { get; set; }
}
```

---

### Q7: Can an interface have a constructor?
**A:** No! Interfaces have no constructors or implementations.

---

### Q8: What is dependency injection?
**A:** Passing dependencies (often interfaces) to a class instead of creating them internally. Makes code testable and flexible.

```csharp
public class UserService
{
    private IDatabase database;

    public UserService(IDatabase db)  // Injected dependency
    {
        database = db;
    }
}

UserService s = new UserService(new SQLDatabase());
```

---

### Q9: Can you use an interface as a type?
**A:** Yes! This is powerful for polymorphism.

```csharp
IAnimal animal = new Dog();
animal.Speak();  // Works!

List<IAnimal> animals = new List<IAnimal>
{
    new Dog(),
    new Cat()
};
```

---

### Q10: What is the "I" prefix convention?
**A:** Interface names start with "I" by convention to distinguish them from classes.

```csharp
ILogger      // Interface
Logger       // Class
IRepository  // Interface
```

## 🚀 Key Takeaways

✅ Interface = Contract defining methods  
✅ Classes implement interfaces  
✅ Multiple interfaces allowed per class  
✅ No implementation in interface  
✅ Enables polymorphism and flexibility  
✅ Use for dependency injection  
✅ Makes code testable and maintainable
