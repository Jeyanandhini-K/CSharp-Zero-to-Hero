# Understanding Project Structure

You created a project in the last lesson. Now let's understand what's *inside* it, so the files stop being mysterious. Knowing this makes everything afterward easier.

---

## What's in a project folder?

After `dotnet new console -o HelloWorld`, your folder looks like this:

```
HelloWorld/
├── HelloWorld.csproj     # the project file (settings)
├── Program.cs            # your code
├── bin/                  # compiled output (auto-generated)
└── obj/                  # build intermediates (auto-generated)
```

- **`Program.cs`** — your actual code. You can add more `.cs` files as your program grows.
- **`HelloWorld.csproj`** — the **project file**. It tells .NET how to build your app.
- **`bin/`** and **`obj/`** — generated automatically when you build. You never edit these by hand, and they shouldn't go into Git (more below).

---

## The `.csproj` file

Open `HelloWorld.csproj`. It's small and written in XML:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

What each line means:

- **`OutputType`** — `Exe` means it builds a runnable program (as opposed to a library).
- **`TargetFramework`** — which .NET version you're building for (`net10.0` here).
- **`ImplicitUsings`** — when `enable`, common namespaces (like `System`) are imported automatically, so you don't need `using System;` at the top. This is why your `Program.cs` could use `Console` without it.
- **`Nullable`** — when `enable`, the compiler helps you avoid null-related bugs (the `string?` you saw earlier is part of this).

As your project grows, the `.csproj` is also where **NuGet package references** go — third-party libraries you add. You'll usually add them with a command rather than editing by hand:

```bash
dotnet add package SomePackageName
```

## The `dotnet` CLI — your daily commands

You'll use the `dotnet` command-line tool constantly. The essentials:

| Command | What it does |
|---|---|
| `dotnet new console -o Name` | Create a new console project in folder `Name` |
| `dotnet run` | Build and run the project in the current folder |
| `dotnet build` | Compile without running |
| `dotnet add package X` | Add a NuGet package |
| `dotnet new list` | Show all available project templates |
| `dotnet --version` | Show the installed SDK version |

Run these from *inside* your project folder.

## Solutions vs projects

- A **project** (`.csproj`) is one buildable thing — one app or one library.
- A **solution** (`.sln`) is a *container* that groups multiple related projects together (for example: a web API project, plus a project of shared code, plus a test project).

For your early learning, **one project is all you need** — don't worry about solutions yet. When you start building larger apps with separate pieces, you'll create one:

```bash
dotnet new sln -o MyApp
```

…and add projects to it. We'll come back to this when it matters.

---

## Namespaces and multiple files

As programs grow, you split code across several `.cs` files. **Namespaces** keep things organized and avoid naming collisions:

```csharp
namespace MyApp.Models;

public class User
{
    public string Name { get; set; }
}
```

A file in one namespace can use a class from another by adding a `using` for it. You'll get plenty of practice with this in the OOP section.

---

## Keep `bin/` and `obj/` out of Git

These folders are regenerated on every build, so they don't belong in version control. The quickest way to get a good ignore file for .NET:

```bash
dotnet new gitignore
```

This drops a ready-made `.gitignore` into your folder that already excludes `bin/`, `obj/`, and other noise. (Your repo already has a `.gitignore` — just make sure those folders are listed.)

---

## Recap

- A project folder has your **`.cs`** code, a **`.csproj`** settings file, and auto-generated **`bin/`** + **`obj/`** folders.
- The **`.csproj`** sets your target .NET version, implicit usings, nullable settings, and package references.
- Use the **`dotnet`** CLI (`new`, `run`, `build`, `add package`) for everyday work.
- A **solution (`.sln`)** groups multiple projects — you don't need one yet.
- Keep **`bin/`** and **`obj/`** out of Git with a `.gitignore`.
