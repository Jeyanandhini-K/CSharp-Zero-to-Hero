# 🗑️ Memory & Garbage Collection in C#

---

## 🧠 How Memory Works in C#

C# manages two areas of memory:

```
┌─────────────────────────────────────────────┐
│                   STACK                      │
│  • Value types (int, bool, struct)           │
│  • Method call frames                        │
│  • Fast allocation & deallocation            │
│  • Auto-cleaned when method returns          │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│                   HEAP                       │
│  • Reference types (class, string, arrays)   │
│  • Longer-lived objects                      │
│  • Managed by the Garbage Collector (GC)     │
│  • Cleaned up automatically (eventually)     │
└─────────────────────────────────────────────┘
```

```csharp
void MyMethod()
{
    int x = 5;              // STACK — gone when method ends
    string s = "hello";     // STACK: reference, HEAP: "hello" string
    var list = new List<int>(); // STACK: reference, HEAP: List object
}   // x cleaned up instantly; list/string cleaned up by GC eventually
```

---

## 🤖 What is the Garbage Collector?

The **GC** automatically finds and frees heap memory that's no longer reachable — you don't have to `free()` or `delete` like C/C++.

```csharp
// You just create objects — GC handles cleanup
var orders = new List<Order>();
for (int i = 0; i < 1000; i++)
    orders.Add(new Order { Id = i });

orders = null;   // nothing references the list anymore
// GC will reclaim all that memory — you don't do anything ✅
```

> 💡 The GC runs in the background. You don't control *when* it runs — but you can help it by writing good code.

## 🏗️ GC Generations — How the GC Works

The GC uses **3 generations** to manage objects efficiently:

```
Generation 0 (Gen 0) — short-lived objects
  • New objects start here
  • Collected very frequently
  • Most objects die here (temp variables, loop objects)

Generation 1 (Gen 1) — medium-lived objects
  • Survived at least one Gen 0 collection
  • Buffer between Gen 0 and Gen 2

Generation 2 (Gen 2) — long-lived objects
  • Survived multiple collections
  • Collected least frequently (expensive!)
  • Static data, caches, singletons live here
```

```csharp
// Gen 0 — short-lived (good!)
for (int i = 0; i < 1000; i++)
{
    var temp = new StringBuilder();   // created and collected quickly
    temp.Append(i);
}

// Gen 2 — long-lived (be careful with large ones)
static List<string> _cache = new();   // lives forever → Gen 2
```

## 📊 Stack vs Heap Performance

```csharp
// Stack allocation — FAST (just move a pointer)
int a = 10;
double b = 3.14;
bool c = true;

// Heap allocation — SLOWER (GC must track it)
var obj = new MyClass();    // heap allocation
var arr = new int[1000];    // heap allocation

// ✅ Minimize heap allocations in hot paths
// ✅ Use structs for small, temporary data
// ✅ Reuse objects instead of creating new ones
```

## 🔢 Span\<T\> — Zero Allocation Magic

`Span<T>` lets you work with slices of memory **without allocating**:

```csharp
// ❌ Old way — creates new string objects (heap allocations)
string text  = "Hello, World!";
string hello = text.Substring(0, 5);   // new string on heap
string world = text.Substring(7, 5);   // another new string

// ✅ New way — Span<T> works on existing memory (zero allocation!)
ReadOnlySpan<char> span  = "Hello, World!";
ReadOnlySpan<char> hello = span[..5];    // no allocation ✅
ReadOnlySpan<char> world = span[7..12];  // no allocation ✅

Console.WriteLine(hello.ToString());   // Hello
Console.WriteLine(world.ToString());   // World

// Span for arrays — slice without copying
int[] numbers = { 1, 2, 3, 4, 5, 6, 7, 8 };
Span<int> middle = numbers.AsSpan()[2..6];   // { 3, 4, 5, 6 }
middle[0] = 99;
Console.WriteLine(numbers[2]);   // 99 — modifies original! ✅
```

## 🚮 Forcing GC (Don't Do This!)

```csharp
// You CAN force GC — but almost never should
GC.Collect();              // force collection now
GC.WaitForPendingFinalizers(); // wait for finalizers

// ❌ Don't do this in production code!
// The GC is smarter than you about when to collect
// Forcing it interrupts your app and hurts performance

// ✅ Only valid use: benchmarks and testing
// to get a clean baseline before measuring
```

## 📏 Measuring Memory

```csharp
// Check how much memory your app is using
long before = GC.GetTotalMemory(forceFullCollection: false);

// Do some work
var data = new List<byte[]>();
for (int i = 0; i < 100; i++)
    data.Add(new byte[1024 * 1024]);   // 100MB

long after = GC.GetTotalMemory(forceFullCollection: false);
Console.WriteLine($"Used: {(after - before) / 1024 / 1024} MB");   // ~100 MB

// GC generation of an object
var obj = new object();
Console.WriteLine(GC.GetGeneration(obj));   // 0 — just created
GC.Collect();
Console.WriteLine(GC.GetGeneration(obj));   // 1 — survived collection
```

## 🌍 Real-World: Memory-Efficient Processing

```csharp
// ❌ Memory-heavy — loads entire file into RAM
public async Task<int> CountWordsHeavy(string path)
{
    string content = await File.ReadAllTextAsync(path);   // entire file in memory!
    return content.Split(' ').Length;
}

// ✅ Memory-efficient — streams line by line
public async Task<int> CountWordsEfficient(string path)
{
    int count = 0;
    await foreach (string line in File.ReadLinesAsync(path))
        count += line.Split(' ').Length;   // one line at a time ✅
    return count;
}

// ✅ Object pooling — reuse instead of allocate
public class ConnectionPool
{
    private readonly Queue<DbConnection> _pool = new();

    public DbConnection Rent()
        => _pool.Count > 0 ? _pool.Dequeue() : new DbConnection();

    public void Return(DbConnection conn)
        => _pool.Enqueue(conn);   // reuse — no new allocation ✅
}
```

## 🏊 ArrayPool\<T\> — Rent and Return Arrays

```csharp
using System.Buffers;

// ❌ Creates a new array every time (heap allocation)
byte[] buffer = new byte[4096];

// ✅ Rent from pool — no allocation if pool has one
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);
try
{
    // use buffer
    int bytesRead = stream.Read(buffer, 0, buffer.Length);
    Process(buffer, bytesRead);
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer);   // return to pool ✅
}
// Use ArrayPool for high-frequency buffer operations (file I/O, networking)
```

## 🎤 Interview Q&A

**Q1: What is the Garbage Collector in C#?**
A: An automatic memory manager that finds and frees heap objects that are no longer reachable — no manual `free()` needed.

**Q2: What is the difference between Stack and Heap?**
A: Stack = value types, fast, auto-cleaned when method returns. Heap = reference types, managed by GC, slower.

**Q3: What are GC generations?**
A: Gen 0 = new short-lived objects (collected most often). Gen 1 = survived one collection. Gen 2 = long-lived objects (collected least).
```csharp
Console.WriteLine(GC.GetGeneration(obj));   // 0, 1, or 2
```

**Q4: Should you call `GC.Collect()` manually?**
A: Almost never — the GC knows better than you when to collect. Forcing it hurts performance. Only use in benchmarks.

**Q5: What is `Span<T>`?**
A: A stack-allocated view over a contiguous region of memory — lets you slice arrays and strings without heap allocation.
```csharp
ReadOnlySpan<char> slice = "Hello World"[..5];   // "Hello" — zero allocation
```

**Q6: What is `ArrayPool<T>`?**
A: A pool of reusable arrays. Rent one instead of allocating, use it, return it. Dramatically reduces GC pressure in high-frequency code.

**Q7: What causes memory leaks in C#?**
A: Objects still referenced (even unintentionally) — event handlers not unsubscribed, static collections that grow forever, not disposing `IDisposable` objects.
```csharp
// Classic leak — event handler keeps subscriber alive
publisher.OnEvent += handler;   // forget to unsubscribe = memory leak
```

**Q8: What is the Large Object Heap (LOH)?**
A: Objects larger than 85KB go to the LOH — collected less frequently and not compacted by default. Avoid large frequent allocations.

**Q9: How do you find memory leaks in C#?**
A: Use dotMemory, PerfView, or Visual Studio Diagnostic Tools. Watch for Gen 2 size growing indefinitely.

**Q10: What is boxing and why is it bad for memory?**
A: Converting a value type to `object` causes a heap allocation (boxing). In tight loops this creates GC pressure.
```csharp
// ❌ Boxing in loop — heap allocation every iteration!
object sum = 0;
for (int i = 0; i < 1_000_000; i++)
    sum = (int)sum + i;   // box, unbox, box, unbox...

// ✅ No boxing — stays on stack
int sum = 0;
for (int i = 0; i < 1_000_000; i++)
    sum += i;
```

## 🚀 Key Takeaways

✅ Stack = fast, auto-cleaned · Heap = GC-managed  
✅ GC automatically frees unreachable objects — you don't free manually  
✅ Gen 0 = short-lived (frequent) · Gen 2 = long-lived (rare)  
✅ Minimize heap allocations in hot paths — use structs, `Span<T>`, pooling  
✅ `Span<T>` = slice memory without allocating  
✅ `ArrayPool<T>` = rent arrays, return when done  
✅ Never call `GC.Collect()` in production  
✅ Memory leaks = forgotten references (events, static lists, undisposed objects)  
✅ Boxing = value type → object → heap allocation — avoid in loops
