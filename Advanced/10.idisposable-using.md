# 🗑️ IDisposable & the `using` Keyword in C#

---

## 🧠 What is Resource Management?

Some objects hold **unmanaged resources** — things the garbage collector can't clean up on its own:

- 📁 File handles
- 🌐 Network connections
- 🗄️ Database connections
- 🖼️ Image/graphics objects
- 🔒 Locks and handles

If you don't release these manually → **memory leaks**, **locked files**, **exhausted connections**. 😱

> 💡 The garbage collector handles **memory** automatically. But it doesn't know how to close a file or disconnect from a database. That's your job.

---

## 🔑 IDisposable — The Interface for Cleanup

Any class that holds unmanaged resources should implement `IDisposable`:

```csharp
public class FileLogger : IDisposable
{
    private StreamWriter _writer;
    private bool _disposed = false;

    public FileLogger(string path)
    {
        _writer = new StreamWriter(path);
    }

    public void Log(string message)
    {
        _writer.WriteLine(message);
    }

    public void Dispose()
    {
        if (!_disposed)
        {
            _writer?.Close();    // release the file handle
            _writer = null;
            _disposed = true;
            Console.WriteLine("FileLogger cleaned up ✅");
        }
    }
}
```

---

## ✅ The `using` Statement — Auto Cleanup

`using` **automatically calls `Dispose()`** when the block ends — even if an exception occurs:

```csharp
// ❌ Without using — easy to forget Dispose!
FileLogger logger = new FileLogger("log.txt");
logger.Log("Starting...");
logger.Dispose();   // what if an exception happens before this? 💥

// ✅ With using — Dispose is ALWAYS called
using (FileLogger logger = new FileLogger("log.txt"))
{
    logger.Log("Starting...");
    logger.Log("Done!");
}   // Dispose() called here automatically ✅
```

---

## 🚀 Modern `using` Declaration (C# 8+)

No curly braces needed — disposes at the end of the **enclosing scope**:

```csharp
// ✅ Cleaner modern syntax
using FileLogger logger = new FileLogger("log.txt");
logger.Log("Starting...");
logger.Log("Done!");
// Dispose() called here when method ends ✅
```

> 💡 Both forms do the same thing. The modern form (no braces) is cleaner for most cases.

## 📁 Real-World: File Operations

```csharp
// Reading a file ✅
using StreamReader reader = new StreamReader("data.txt");
string content = reader.ReadToEnd();
Console.WriteLine(content);
// reader.Dispose() called automatically

// Writing a file ✅
using StreamWriter writer = new StreamWriter("output.txt");
writer.WriteLine("Hello, World!");
writer.WriteLine("Second line");
// writer.Dispose() called automatically

// Even simpler — File helpers handle using internally
string text = File.ReadAllText("data.txt");       // ✅
File.WriteAllText("output.txt", "Hello!");        // ✅
string[] lines = File.ReadAllLines("data.txt");   // ✅
```

## 🌐 Real-World: HttpClient

```csharp
// ⚠️ Don't put HttpClient in a using block — it's meant to be reused!
// Create once, reuse many times
private static readonly HttpClient _client = new HttpClient();

public async Task<string> GetDataAsync(string url)
{
    // HttpClient manages its own connections
    string response = await _client.GetStringAsync(url);
    return response;
}
```

> 🚨 `HttpClient` is a famous exception — **don't** `using` it. Create one instance and reuse it throughout your app.

## 🗄️ Real-World: Database Connections

```csharp
using System.Data.SqlClient;

public async Task<List<string>> GetNamesAsync()
{
    var names = new List<string>();

    // Connection is opened and closed automatically ✅
    using SqlConnection conn = new SqlConnection("your-connection-string");
    await conn.OpenAsync();

    using SqlCommand cmd = new SqlCommand("SELECT Name FROM Users", conn);
    using SqlDataReader reader = await cmd.ExecuteReaderAsync();

    while (await reader.ReadAsync())
    {
        names.Add(reader.GetString(0));
    }

    return names;
}   // conn, cmd, reader all disposed here ✅
```

## 🔒 Dispose Pattern — The Full Implementation

For production classes, implement the full dispose pattern:

```csharp
public class ResourceHolder : IDisposable
{
    private bool _disposed = false;

    // Managed resources (other IDisposable objects)
    private StreamWriter _writer;

    // Unmanaged resources (raw handles — rare)
    private IntPtr _handle;

    public ResourceHolder()
    {
        _writer = new StreamWriter("log.txt");
    }

    // Called by user code: using (var r = new ResourceHolder()) { }
    public void Dispose()
    {
        Dispose(disposing: true);
        GC.SuppressFinalize(this);   // no need for finalizer to run
    }

    // Called by GC if user forgot to Dispose
    ~ResourceHolder()
    {
        Dispose(disposing: false);
    }

    protected virtual void Dispose(bool disposing)
    {
        if (!_disposed)
        {
            if (disposing)
            {
                _writer?.Dispose();   // clean up managed resources
            }
            // clean up unmanaged resources here (if any)
            _disposed = true;
        }
    }
}
```

> 💡 For most code you write, the **simple version** (just implement `Dispose()`) is enough. The full pattern is for library authors and complex scenarios.

## ⚡ IAsyncDisposable — Async Cleanup

For async cleanup (closing a database async, etc.), use `IAsyncDisposable`:

```csharp
public class AsyncLogger : IAsyncDisposable
{
    private StreamWriter _writer = new StreamWriter("log.txt");

    public async Task LogAsync(string msg)
    {
        await _writer.WriteLineAsync(msg);
    }

    public async ValueTask DisposeAsync()
    {
        await _writer.FlushAsync();
        await _writer.DisposeAsync();
        Console.WriteLine("AsyncLogger cleaned up ✅");
    }
}

// Use with await using
await using AsyncLogger logger = new AsyncLogger();
await logger.LogAsync("Hello async!");
// DisposeAsync() called automatically ✅
```

## 📋 Summary Table

| Concept | Syntax | What it does |
|---|---|---|
| `IDisposable` | `public void Dispose()` | Interface for cleanup |
| `using` block | `using (var x = new X()) { }` | Auto-calls Dispose on exit |
| `using` declaration | `using var x = new X();` | Disposes at end of scope |
| `await using` | `await using var x = new X();` | Async dispose |
| `IAsyncDisposable` | `public ValueTask DisposeAsync()` | Async cleanup interface |
| `GC.SuppressFinalize` | `GC.SuppressFinalize(this)` | Skip finalizer after Dispose |

## 🎤 Interview Q&A

**Q1: What is IDisposable?**
A: An interface with a `Dispose()` method. Implement it on classes that hold resources the GC can't clean up (files, connections, handles).

**Q2: What does the `using` statement do?**
A: Automatically calls `Dispose()` when the block exits — even if an exception occurs.
```csharp
using StreamReader r = new StreamReader("file.txt");
// r.Dispose() called at end of scope
```

**Q3: What's the difference between `using` block and `using` declaration?**
A: `using` block (with `{}`) disposes at the closing brace. `using` declaration (no braces, C# 8+) disposes at the end of the enclosing method. Same result, declaration is cleaner.

**Q4: What resources need manual disposal?**
A: File streams, database connections, network connections, graphics objects, mutexes — anything not purely managed memory.

**Q5: Does the garbage collector call Dispose?**
A: Not automatically. You call `Dispose()` (or use `using`). The GC calls the **finalizer** as a safety net, but you shouldn't rely on it.

**Q6: What is `IAsyncDisposable`?**
A: Like `IDisposable` but for async cleanup. Implement `DisposeAsync()` and use `await using`.

**Q7: Why should HttpClient NOT be in a using block?**
A: It's designed to be reused. Creating and disposing it frequently exhausts socket connections. Create one static instance per app.

**Q8: What is GC.SuppressFinalize?**
A: Tells the GC "don't run the finalizer — we already cleaned up in Dispose". Prevents double-cleanup.

**Q9: What happens if you forget to Dispose?**
A: Resource leaks — files stay locked, database connections stay open, memory isn't freed. Your app slows down or crashes.

**Q10: What is the finalizer (~ClassName)?**
A: A safety net called by the GC if the user forgot to Dispose. Slower than Dispose because it runs on the GC thread and delays collection.

## 🚀 Key Takeaways

✅ Some resources need manual cleanup — GC won't do it for you  
✅ Implement `IDisposable` on classes that hold files, connections, handles  
✅ `using` block / declaration auto-calls `Dispose()` — always use it  
✅ `using` protects you even when exceptions occur  
✅ Modern syntax: `using var x = new X();` (no braces, C# 8+)  
✅ `IAsyncDisposable` + `await using` for async cleanup  
✅ **Never** put `HttpClient` in a `using` block — reuse it  
✅ Forgetting to Dispose = memory leaks + locked files 😱
