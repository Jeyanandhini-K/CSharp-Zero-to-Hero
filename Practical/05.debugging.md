# 🐛 Debugging in C#

---

## 🧠 What is Debugging?

**Debugging** is finding and fixing bugs — the inevitable part of writing code. The better you debug, the faster you grow as a developer.

```csharp
// Bug: why does this print 0?
int total = 0;
for (int i = 1; i < 10; i += 2)
    total += i;
Console.WriteLine(total);   // 25, not 0 — but WHY?

// A debugger lets you PAUSE here and inspect every value
// No more guessing 🎯
```

> 💡 `Console.WriteLine` everywhere is how beginners debug. **Breakpoints** are how professionals debug. Let's upgrade you.

---

## 🔴 Breakpoints — Pause Code Mid-Execution

A **breakpoint** tells the debugger: *"Stop here. Let me look around."*

**In VS Code:**
1. Click the **left gutter** (grey area left of line numbers) → red dot appears 🔴
2. Press **F5** to start debugging
3. Code pauses at your breakpoint
4. Hover over any variable to see its value

**In Visual Studio:**
- Same — click the gutter or press **F9** to toggle a breakpoint
- **F5** starts debugging
- **Shift+F5** stops

---

## ⏭️ Stepping Through Code

Once paused at a breakpoint:

| Key (VS Code) | Key (VS) | What it does |
|---|---|---|
| **F10** | F10 | **Step Over** — run next line, don't go into methods |
| **F11** | F11 | **Step Into** — go inside the method being called |
| **Shift+F11** | Shift+F11 | **Step Out** — finish current method, go back to caller |
| **F5** | F5 | **Continue** — run until next breakpoint |
| **Shift+F5** | Shift+F5 | **Stop** debugging |

```csharp
int result = Add(5, 3);   // F10 runs this line; F11 steps INTO Add()
Console.WriteLine(result);

int Add(int a, int b) => a + b;
```

## 🔍 The Watch Window & Locals

When paused at a breakpoint:

- **Locals** panel — automatically shows all variables in scope and their current values
- **Watch** panel — type any expression (`user.Name`, `list.Count`, `a + b`) and see it live
- **Hover** over any variable in code → tooltip shows its value

```csharp
var users = GetUsers();       // ← Breakpoint here
var active = users.Where(u => u.IsActive).ToList();

// In Locals: users = List<User>(15 items)
// In Watch:  users.Count = 15, active.Count = ?  ← not calculated yet
```

## 🎯 Conditional Breakpoints

Don't want to step through 10,000 loop iterations? Set a condition:

**VS Code / VS:**
- Right-click a breakpoint → **Edit Breakpoint** → add a condition

```csharp
for (int i = 0; i < 10000; i++)
{
    Process(data[i]);   // ← Breakpoint condition: i == 9999
}                       // Only pauses on the LAST iteration
```

> 🔥 Conditional breakpoints are a superpower — use them for loops and large datasets.

---

## 📋 The Debug Output Window

`Debug.WriteLine` prints to the **Output** window — not the console — great for quick inspection without changing your UI:

```csharp
using System.Diagnostics;

Debug.WriteLine($"User loaded: {user.Name}");       // only shows in debug mode
Debug.WriteLine($"Items in cart: {cart.Count}");
Debug.Assert(user != null, "User should not be null!");  // crashes in debug if false
```

---

## 💡 Exception Debugging

When an exception occurs, the debugger can pause right at the throw:

**VS Code:** Run → **Add Configuration** → set `stopOnEntry: true`
**VS:** Debug → **Windows → Exception Settings** → check `Common Language Runtime Exceptions`

```csharp
try
{
    var result = int.Parse("not a number");   // ← Debugger pauses HERE
}
catch (FormatException ex)
{
    // Now inspect ex.Message, ex.StackTrace in the debugger
    Console.WriteLine(ex.Message);
}
```

## 🪵 Logging as Debugging

For issues you can't reproduce in a debugger (production bugs, async timing issues), structured logs are your best friend:

```csharp
using Microsoft.Extensions.Logging;

public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(ILogger<OrderService> logger) => _logger = logger;

    public async Task ProcessOrderAsync(int orderId)
    {
        _logger.LogInformation("Processing order {OrderId}", orderId);

        try
        {
            var order = await GetOrderAsync(orderId);
            _logger.LogDebug("Order found: {Order}", order);

            await SaveAsync(order);
            _logger.LogInformation("Order {OrderId} saved successfully", orderId);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed to process order {OrderId}", orderId);
            throw;
        }
    }
}
```

## 🕵️ Common Bugs & How to Find Them

| Bug | Symptom | Debug strategy |
|---|---|---|
| `NullReferenceException` | Crash on `.Property` | Check which object is null in Locals |
| Off-by-one | Loop runs too many/few times | Conditional breakpoint at boundary |
| Wrong value | Calculation returns unexpected result | Watch window on the expression |
| Async bug | Method returns before awaited | Check you used `await` everywhere |
| Race condition | Bug only happens sometimes | Add logging; use thread-safe collections |
| Infinite loop | App freezes | Pause debugger manually (pause button) |

## 🌍 Real-World: Debug Checklist

```
When something goes wrong:

1. 🔴 Set a breakpoint BEFORE the bug happens
2. 👀 Check Locals — which value is wrong?
3. 🔍 Add it to Watch — track it across lines
4. ⏭️ Step through line by line — where does it go wrong?
5. 🤔 Ask: is it null? wrong type? off by one? not awaited?
6. 📋 Read the full exception message AND stack trace
7. 🌐 Google the exact exception + method name if stuck
```

## 🎤 Interview Q&A

**Q1: What is a breakpoint?**
A: A marker that tells the debugger to pause execution at that line so you can inspect variables.

**Q2: What is the difference between Step Over and Step Into?**
A: Step Over (F10) runs the line without going inside called methods. Step Into (F11) enters the method being called.

**Q3: What is the Watch window?**
A: A panel in the debugger where you type any expression and see its current value while paused.

**Q4: What is a conditional breakpoint?**
A: A breakpoint that only triggers when a condition is true — e.g. `i == 999` to stop on the 1000th loop iteration.

**Q5: How do you debug a NullReferenceException?**
A: Set a breakpoint before the crash, check Locals to find which variable is null, trace back to where it should have been assigned.

**Q6: What is `Debug.Assert`?**
A: A statement that crashes the app (in Debug mode only) if the condition is false — great for catching wrong assumptions early.
```csharp
Debug.Assert(user != null, "User must not be null here!");
```

**Q7: How do you debug production issues you can't reproduce locally?**
A: Use structured logging (Serilog, ILogger). Log inputs, outputs, and errors so you have a trace of what happened.

**Q8: What is the call stack?**
A: A list of method calls that led to the current line. In the debugger, the Call Stack panel shows you the full path from `Main` to where you are now.

**Q9: How do you pause on an exception before it's caught?**
A: In VS: Debug → Exception Settings → check CLR exceptions. The debugger pauses at the throw, not the catch.

**Q10: What is hot reload?**
A: A .NET feature that lets you change code **while the app is running** without restarting — speeds up debugging loops significantly.

## 🚀 Key Takeaways

✅ Breakpoints pause code — use them instead of `Console.WriteLine` everywhere  
✅ F10 = Step Over · F11 = Step Into · F5 = Continue  
✅ Locals panel — all variables in scope, automatically  
✅ Watch panel — track any expression live  
✅ Conditional breakpoints — pause only when you need to  
✅ `Debug.Assert` — catch wrong assumptions in debug mode  
✅ Read the **full** exception message AND stack trace — it tells you everything  
✅ Use structured logging for bugs you can't reproduce locally  
✅ The call stack shows you HOW you got to the current line
