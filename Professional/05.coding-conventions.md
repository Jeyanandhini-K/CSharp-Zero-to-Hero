# 📏 Coding Conventions in C#

---

## 🧠 Why Conventions Matter

Code is read far more than it's written. Conventions make code **predictable** — a teammate (or future you) can read it without decoding it first.

```csharp
// ❌ No conventions — what does this even do?
public class m { public int x(int a,int b){return a+b;} }

// ✅ With conventions — crystal clear
public class Calculator
{
    public int Add(int firstNumber, int secondNumber)
        => firstNumber + secondNumber;
}
```

> 💡 Microsoft publishes official C# naming guidelines. Real teams follow them. Interviewers notice when you don't.

## 🔤 Naming Rules — The Big One

| What | Convention | Example |
|---|---|---|
| Classes | PascalCase | `OrderService`, `UserRepository` |
| Interfaces | `I` + PascalCase | `IOrderService`, `ILogger` |
| Methods | PascalCase | `GetByIdAsync`, `PlaceOrder` |
| Properties | PascalCase | `FirstName`, `TotalAmount` |
| Public fields | PascalCase | `MaxRetries` |
| Private fields | `_` + camelCase | `_orderRepository`, `_logger` |
| Local variables | camelCase | `userName`, `itemCount` |
| Parameters | camelCase | `userId`, `emailAddress` |
| Constants | PascalCase | `MaxRetryCount`, `DefaultTimeout` |
| Enums | PascalCase | `OrderStatus.Pending` |
| Async methods | `Async` suffix | `GetUserAsync`, `SaveAsync` |

```csharp
public class UserService   // PascalCase class
{
    private readonly IUserRepository _repo;   // _camelCase private field
    private const int MaxLoginAttempts = 5;   // PascalCase constant

    public UserService(IUserRepository repo) => _repo = repo;   // camelCase param

    public async Task<User?> GetUserAsync(int userId)   // PascalCase method, Async suffix
    {
        int attemptCount = 0;   // camelCase local variable
        return await _repo.GetByIdAsync(userId);
    }
}
```

## 📐 Formatting Rules

```csharp
// ✅ Braces on new lines (Allman style — C# standard)
public class Order
{
    public int Id { get; set; }
}

// ✅ One statement per line
int x = 5;
int y = 10;

// ❌ Multiple statements on one line
int x = 5; int y = 10;

// ✅ Spaces around operators
int total = price * quantity + tax;

// ❌ No spaces
int total=price*quantity+tax;

// ✅ Blank line between methods
public void MethodOne() { }

public void MethodTwo() { }

// ✅ Max line length ~120 characters — wrap long lines
var result = await _orderService
    .PlaceOrderAsync(order, cancellationToken);
```

## 🛠️ var — When to Use It

```csharp
// ✅ Use var when the type is obvious from the right side
var list    = new List<string>();       // obviously List<string>
var service = new OrderService();      // obviously OrderService
var name    = "Asha";                  // obviously string

// ❌ Don't use var when the type is unclear
var result = GetData();     // what type is result? 😕 — use explicit type

// ✅ Always explicit for primitives (clearer)
int count   = 0;
bool isActive = true;
string name = "Asha";
```

## 📁 File & Project Conventions

```
✅ One class per file
✅ File name matches class name exactly — UserService.cs for class UserService
✅ Namespace matches folder structure — MyApp.Services for Services/ folder
✅ Group related files in folders — Controllers/, Services/, Models/, Repositories/

// Standard folder structure
MyApp/
├── Controllers/    UserController.cs, OrderController.cs
├── Services/       UserService.cs, OrderService.cs
├── Repositories/   SqlUserRepo.cs, SqlOrderRepo.cs
├── Models/         User.cs, Order.cs, Product.cs
└── Program.cs
```

## 💬 Comments — Less is More

```csharp
// ✅ Comment WHY, not WHAT (the code says what — you say why)
// Retry 3 times because the payment API occasionally times out on first call
for (int i = 0; i < 3; i++) { ... }

// ❌ Useless — the code already says this
// Increment counter by 1
counter++;

// ✅ XML doc comments on public APIs
/// <summary>
/// Places an order and sends a confirmation email.
/// </summary>
/// <param name="order">The order to place. Must have a positive total.</param>
/// <returns>True if order was placed successfully.</returns>
/// <exception cref="ArgumentException">Thrown when order total is zero or negative.</exception>
public async Task<bool> PlaceOrderAsync(Order order) { ... }
```

## ✨ Modern C# — Write It the Clean Way

```csharp
// ✅ Expression body for simple members
public string FullName => $"{FirstName} {LastName}";
public int Add(int a, int b) => a + b;

// ✅ Primary constructors (C# 12)
public class OrderService(IOrderRepository repo, ILogger<OrderService> logger)
{
    public async Task SaveAsync(Order o) => await repo.SaveAsync(o);
}

// ✅ Object initializers
var order = new Order
{
    Id    = 1,
    Total = 999.99m,
    Email = "asha@test.com"
};

// ✅ Collection expressions (C# 12)
int[] nums = [1, 2, 3, 4, 5];
List<string> names = ["Asha", "Ravi", "Priya"];

// ✅ Null coalescing and conditional
string name = user?.Name ?? "Anonymous";
int length  = text?.Length ?? 0;

// ✅ Pattern matching over type checking
if (shape is Circle { Radius: > 10 } c)
    Console.WriteLine($"Big circle: {c.Radius}");
```

## 🚫 Common Bad Habits to Break

```csharp
// ❌ Magic numbers — what does 86400 mean?
if (seconds > 86400) { }

// ✅ Named constant
const int SecondsPerDay = 86400;
if (seconds > SecondsPerDay) { }

// ❌ Catching everything silently
try { DoSomething(); }
catch (Exception) { }   // swallows all errors — terrible!

// ✅ Catch specifically and log
try { DoSomething(); }
catch (SqlException ex) { _logger.LogError(ex, "Database error"); throw; }

// ❌ Returning null from collections
public List<User> GetUsers()
{
    if (notFound) return null;   // forces null checks on every caller
}

// ✅ Return empty collection
public List<User> GetUsers()
{
    if (notFound) return [];     // no null check needed
}

// ❌ Boolean parameter — what does 'true' mean here?
ProcessOrder(order, true);

// ✅ Use an enum or named parameter
ProcessOrder(order, sendEmail: true);
```

## 🔧 EditorConfig — Enforce Conventions Automatically

Add a `.editorconfig` to your project root — VS Code and VS enforce it automatically:

```ini
root = true

[*.cs]
indent_style             = space
indent_size              = 4
end_of_line              = crlf
charset                  = utf-8-bom
trim_trailing_whitespace = true
insert_final_newline     = true

# naming
dotnet_naming_rule.private_fields_underscore.symbols  = private_fields
dotnet_naming_rule.private_fields_underscore.style    = underscore_camel
dotnet_naming_symbols.private_fields.applicable_kinds = field
dotnet_naming_symbols.private_fields.applicable_accessibilities = private
dotnet_naming_style.underscore_camel.required_prefix  = _
dotnet_naming_style.underscore_camel.capitalization   = camel_case
```

## 🎤 Interview Q&A

**Q1: What naming convention do C# classes use?**
A: PascalCase — `OrderService`, `UserRepository`.

**Q2: How should private fields be named?**
A: `_camelCase` — underscore prefix, then camelCase: `_orderRepository`, `_logger`.

**Q3: What prefix do interfaces use?**
A: `I` — `IOrderService`, `ILogger`, `IRepository`.

**Q4: When should you use `var`?**
A: When the type is obvious from the right-hand side. Avoid when the type would be unclear to a reader.

**Q5: What is the Allman brace style?**
A: Opening brace on a new line — the standard in C#:
```csharp
public class Foo
{
    public void Bar() { }
}
```

**Q6: How should async methods be named?**
A: With an `Async` suffix — `GetUserAsync`, `SaveAsync`, `DeleteAsync`.

**Q7: What is a magic number and why is it bad?**
A: A hardcoded literal (`86400`, `404`, `3`) with no explanation. Replace with a named constant — `SecondsPerDay`, `HttpNotFound`, `MaxRetries`.

**Q8: What should comments explain?**
A: The WHY — reasoning, decisions, non-obvious behaviour. Not the WHAT — the code already says that.

**Q9: What is `.editorconfig`?**
A: A file that enforces code style rules (indentation, naming, braces) automatically in VS Code and Visual Studio.

**Q10: What is the convention for constants?**
A: PascalCase — `MaxRetryCount`, `DefaultTimeout`, `SecondsPerDay`.

## 🚀 Key Takeaways

✅ Classes, methods, properties → **PascalCase**  
✅ Private fields → **`_camelCase`**  
✅ Local variables, parameters → **camelCase**  
✅ Interfaces → **`I` prefix**  
✅ Async methods → **`Async` suffix**  
✅ One class per file, filename = classname  
✅ Use `var` only when type is obvious from the right side  
✅ Comments explain **WHY**, not what  
✅ No magic numbers — use named constants  
✅ Add `.editorconfig` to enforce conventions automatically
