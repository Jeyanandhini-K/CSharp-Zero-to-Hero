# 🏛️ SOLID Principles in C#

---

## 🧠 What is SOLID?

**SOLID** is 5 principles that make your code easier to read, test, change, and grow. Every professional C# developer lives by them.

| Letter | Principle | One-line summary |
|---|---|---|
| **S** | Single Responsibility | One class = one job |
| **O** | Open/Closed | Open to extend, closed to modify |
| **L** | Liskov Substitution | Subclasses must not break parent behaviour |
| **I** | Interface Segregation | Small, focused interfaces over fat ones |
| **D** | Dependency Inversion | Depend on abstractions, not concrete classes |

---

## 🅢 Single Responsibility Principle

> **"A class should have only one reason to change."**

```csharp
// ❌ BAD — one class doing three jobs
public class UserManager
{
    public void CreateUser(User user) { /* save to DB */ }
    public void SendWelcomeEmail(User user) { /* send email */ }
    public string GenerateReport(User user) { /* build report */ }
}

// ✅ GOOD — each class has one job
public class UserRepository   { public void Save(User user) { } }
public class EmailService     { public void SendWelcome(User user) { } }
public class UserReportBuilder{ public string Build(User user) => ""; }
```

> 💡 Ask yourself: *"What is the ONE reason this class would need to change?"* If there are two reasons — split it.

## 🅞 Open/Closed Principle

> **"Open for extension, closed for modification."**

Add new behaviour by **adding new code**, not by changing existing code.

```csharp
// ❌ BAD — adding a new discount means editing this method every time
public decimal GetDiscount(string customerType)
{
    if (customerType == "Regular") return 0.05m;
    if (customerType == "VIP")     return 0.10m;
    if (customerType == "Staff")   return 0.20m;   // ← had to open and modify!
    return 0;
}

// ✅ GOOD — add new discount types without touching existing code
public interface IDiscountStrategy
{
    decimal GetDiscount();
}

public class RegularDiscount : IDiscountStrategy { public decimal GetDiscount() => 0.05m; }
public class VipDiscount     : IDiscountStrategy { public decimal GetDiscount() => 0.10m; }
public class StaffDiscount   : IDiscountStrategy { public decimal GetDiscount() => 0.20m; }

// New customer type? Just add a new class — no existing code touched ✅
public class NewJoinerDiscount : IDiscountStrategy { public decimal GetDiscount() => 0.15m; }
```

## 🅛 Liskov Substitution Principle

> **"Subclasses must be replaceable for their parent class without breaking things."**

```csharp
// ❌ BAD — Square breaks the Rectangle contract
public class Rectangle
{
    public virtual int Width  { get; set; }
    public virtual int Height { get; set; }
    public int Area() => Width * Height;
}

public class Square : Rectangle
{
    public override int Width  { set { base.Width = base.Height = value; } }
    public override int Height { set { base.Width = base.Height = value; } }
}

// This breaks! Setting Width and Height independently doesn't work on a Square
Rectangle r = new Square();
r.Width  = 4;
r.Height = 5;
Console.WriteLine(r.Area());   // ❌ Expected 20, got 25!

// ✅ GOOD — separate abstractions that truly model each shape
public abstract class Shape  { public abstract int Area(); }
public class Rectangle : Shape
{
    public int Width { get; set; }
    public int Height { get; set; }
    public override int Area() => Width * Height;
}
public class Square : Shape
{
    public int Side { get; set; }
    public override int Area() => Side * Side;
}
```

> 💡 If your subclass has to throw `NotImplementedException` or override something in a way that surprises callers — LSP is violated.

## 🅘 Interface Segregation Principle

> **"Clients should not be forced to implement interfaces they don't use."**

```csharp
// ❌ BAD — one fat interface forces classes to implement things they don't need
public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
}

public class Robot : IWorker
{
    public void Work() { /* fine */ }
    public void Eat()  { throw new NotImplementedException(); }   // ❌ robots don't eat!
    public void Sleep(){ throw new NotImplementedException(); }   // ❌ robots don't sleep!
}

// ✅ GOOD — small, focused interfaces
public interface IWorkable  { void Work(); }
public interface IFeedable  { void Eat();  }
public interface ISleepable { void Sleep(); }

public class Human : IWorkable, IFeedable, ISleepable
{
    public void Work()  { }
    public void Eat()   { }
    public void Sleep() { }
}

public class Robot : IWorkable
{
    public void Work() { }   // only implements what it actually does ✅
}
```

## 🅓 Dependency Inversion Principle

> **"Depend on abstractions (interfaces), not concrete implementations."**

```csharp
// ❌ BAD — UserService is tightly coupled to SqlDatabase
public class UserService
{
    private SqlDatabase _db = new SqlDatabase();   // ← hard dependency!

    public void Save(User user) => _db.Insert(user);
}
// Can't test without a real SQL database. Can't swap to MongoDB. 😢

// ✅ GOOD — UserService depends on an abstraction
public interface IUserRepository
{
    void Save(User user);
}

public class SqlUserRepository   : IUserRepository { public void Save(User u) { /* SQL */ } }
public class MongoUserRepository : IUserRepository { public void Save(User u) { /* Mongo */ } }
public class FakeUserRepository  : IUserRepository { public void Save(User u) { /* tests */ } }

public class UserService
{
    private readonly IUserRepository _repo;

    public UserService(IUserRepository repo) => _repo = repo;   // injected ✅

    public void Save(User user) => _repo.Save(user);
}

// Now you can swap the database with zero changes to UserService ✅
var service = new UserService(new SqlUserRepository());
var testSvc = new UserService(new FakeUserRepository());
```

## 🌍 All 5 Together — Real Example

```csharp
// S — one job each
public class OrderValidator  { public bool IsValid(Order o) => o.Total > 0; }
public class OrderRepository { public async Task SaveAsync(Order o) { } }
public class OrderNotifier   { public async Task NotifyAsync(Order o) { } }

// O — extend without modifying
public interface IShippingStrategy { decimal Calculate(Order o); }
public class StandardShipping : IShippingStrategy { public decimal Calculate(Order o) => 50; }
public class ExpressShipping  : IShippingStrategy { public decimal Calculate(Order o) => 150; }

// I — focused interfaces
public interface IOrderReader { Task<Order?> GetByIdAsync(int id); }
public interface IOrderWriter { Task SaveAsync(Order order); }

// D — depend on interfaces
public class OrderService
{
    private readonly IOrderReader _reader;
    private readonly IOrderWriter _writer;
    private readonly OrderValidator _validator;
    private readonly OrderNotifier _notifier;

    public OrderService(IOrderReader reader, IOrderWriter writer,
                        OrderValidator validator, OrderNotifier notifier)
    {
        _reader = reader; _writer = writer;
        _validator = validator; _notifier = notifier;
    }

    public async Task PlaceAsync(Order order)
    {
        if (!_validator.IsValid(order))
            throw new ArgumentException("Invalid order");

        await _writer.SaveAsync(order);
        await _notifier.NotifyAsync(order);
    }
}
```

## 📋 Summary Table

| Principle | Problem it solves | Key question to ask |
|---|---|---|
| **S** — Single Responsibility | Classes that do too much | "What is the ONE reason to change?" |
| **O** — Open/Closed | Modifying existing code to add features | "Can I add this without touching old code?" |
| **L** — Liskov Substitution | Subclasses that break parent behaviour | "Can I swap this for the parent without surprise?" |
| **I** — Interface Segregation | Fat interfaces with unused methods | "Does every implementer use every method?" |
| **D** — Dependency Inversion | Hard-coded dependencies | "Am I depending on an interface or a concrete class?" |

## 🎤 Interview Q&A

**Q1: What does SOLID stand for?**
A: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion.

**Q2: What is the Single Responsibility Principle?**
A: A class should have only one reason to change — one job, one responsibility.

**Q3: What is the Open/Closed Principle?**
A: Code should be open for extension (add new behaviour) but closed for modification (don't change existing code).

**Q4: What is the Liskov Substitution Principle?**
A: A subclass must be usable wherever its parent is used, without breaking the expected behaviour.

**Q5: What is the Interface Segregation Principle?**
A: Don't force classes to implement interfaces they don't use. Split fat interfaces into small, focused ones.

**Q6: What is the Dependency Inversion Principle?**
A: High-level modules should depend on abstractions (interfaces), not concrete implementations.

**Q7: Which SOLID principle is most related to unit testing?**
A: DIP — depending on interfaces makes classes mockable and independently testable.

**Q8: How does OCP relate to design patterns?**
A: Strategy, Factory, and Decorator patterns are common ways to achieve OCP — adding behaviour through new classes.

**Q9: What's a sign of SRP violation?**
A: A class with many methods covering unrelated concerns, or a class that changes for multiple different reasons.

**Q10: Can you over-apply SOLID?**
A: Yes — too many tiny interfaces and classes creates unnecessary complexity. Apply SOLID where it adds value, not mechanically everywhere.

## 🚀 Key Takeaways

✅ **S** — one class, one job — easy to test and change  
✅ **O** — add features by adding code, not editing it  
✅ **L** — subclasses must honour their parent's contract  
✅ **I** — small focused interfaces, no forced `NotImplementedException`  
✅ **D** — inject interfaces, not concrete classes  
✅ DIP makes code **testable** — you can mock everything  
✅ OCP makes code **extensible** — new features don't break old ones  
✅ SRP makes code **readable** — one clear purpose per class
