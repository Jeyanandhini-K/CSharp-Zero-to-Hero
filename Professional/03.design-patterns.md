# 🎨 Design Patterns in C#

---

## 🧠 What are Design Patterns?

Design patterns are **proven, reusable solutions** to problems that keep coming up in software. Not copy-paste code — blueprints for solving a category of problem.

```
Gang of Four (GoF) → 23 classic patterns → 3 categories:

🏗️ Creational  — how objects are created
🔗 Structural  — how objects are composed
🔄 Behavioural — how objects communicate
```

> 💡 You don't need all 23. Master these 6 and you'll handle 90% of real situations.

## 1️⃣ Singleton — One Instance Only 🏗️

**Problem:** Some objects should only ever exist once — a config loader, a logger, a connection pool.

```csharp
// ✅ Thread-safe Singleton using Lazy<T>
public class AppConfig
{
    private static readonly Lazy<AppConfig> _instance =
        new(() => new AppConfig());

    public static AppConfig Instance => _instance.Value;

    public string DatabaseUrl { get; private set; }
    public string ApiKey      { get; private set; }

    private AppConfig()   // private constructor — no one can do new AppConfig()
    {
        DatabaseUrl = "Server=localhost;Database=MyApp";
        ApiKey      = "abc-123-secret";
    }
}

// Usage — same instance everywhere
AppConfig config = AppConfig.Instance;
Console.WriteLine(config.DatabaseUrl);
```

> ⚠️ Singletons are powerful but overused. Hard to test. Use only when truly one instance makes sense.

## 2️⃣ Factory — Delegate Object Creation 🏗️

**Problem:** You need to create objects but want to hide the creation logic — or choose which type to create at runtime.

```csharp
// Factory Method
public abstract class Notification
{
    public abstract void Send(string message);
}

public class EmailNotification : Notification
{
    public override void Send(string msg) => Console.WriteLine($"📧 Email: {msg}");
}

public class SmsNotification : Notification
{
    public override void Send(string msg) => Console.WriteLine($"📱 SMS: {msg}");
}

public class PushNotification : Notification
{
    public override void Send(string msg) => Console.WriteLine($"🔔 Push: {msg}");
}

// Factory — decides WHICH type to create
public static class NotificationFactory
{
    public static Notification Create(string type) => type switch
    {
        "email" => new EmailNotification(),
        "sms"   => new SmsNotification(),
        "push"  => new PushNotification(),
        _       => throw new ArgumentException($"Unknown type: {type}")
    };
}

// Usage — caller doesn't know or care which class it gets
Notification n = NotificationFactory.Create("email");
n.Send("Your order is confirmed!");
```

## 3️⃣ Repository — Abstract Data Access 🔗

**Problem:** Business logic shouldn't know or care whether data comes from SQL, MongoDB, or a file.

```csharp
// The abstraction
public interface IProductRepository
{
    Task<Product?> GetByIdAsync(int id);
    Task<List<Product>> GetAllAsync();
    Task SaveAsync(Product product);
    Task DeleteAsync(int id);
}

// SQL implementation
public class SqlProductRepository : IProductRepository
{
    private readonly string _connectionString;
    public SqlProductRepository(string cs) => _connectionString = cs;

    public async Task<Product?> GetByIdAsync(int id)
    {
        // real SQL query here
        return null;
    }
    public async Task<List<Product>> GetAllAsync() => new();
    public async Task SaveAsync(Product product) { }
    public async Task DeleteAsync(int id) { }
}

// Fake for tests ✅
public class FakeProductRepository : IProductRepository
{
    private readonly List<Product> _data = new();
    public Task<Product?> GetByIdAsync(int id)    => Task.FromResult(_data.FirstOrDefault(p => p.Id == id));
    public Task<List<Product>> GetAllAsync()       => Task.FromResult(_data);
    public Task SaveAsync(Product p)               { _data.Add(p); return Task.CompletedTask; }
    public Task DeleteAsync(int id)                { _data.RemoveAll(p => p.Id == id); return Task.CompletedTask; }
}

// Service depends on the interface — swappable ✅
public class ProductService
{
    private readonly IProductRepository _repo;
    public ProductService(IProductRepository repo) => _repo = repo;

    public async Task<List<Product>> GetAllAsync() => await _repo.GetAllAsync();
}
```

## 4️⃣ Strategy — Swap Algorithms at Runtime 🔄

**Problem:** You need to choose between different algorithms or behaviours at runtime without messy `if/else`.

```csharp
// Strategy interface
public interface ISortStrategy
{
    void Sort(List<int> data);
}

// Concrete strategies
public class BubbleSort : ISortStrategy
{
    public void Sort(List<int> data)
    {
        Console.WriteLine("Sorting with Bubble Sort...");
        // bubble sort logic
    }
}

public class QuickSort : ISortStrategy
{
    public void Sort(List<int> data)
    {
        Console.WriteLine("Sorting with Quick Sort...");
        data.Sort();   // simplified
    }
}

// Context — uses whichever strategy is injected
public class Sorter
{
    private ISortStrategy _strategy;

    public Sorter(ISortStrategy strategy) => _strategy = strategy;

    public void SetStrategy(ISortStrategy strategy) => _strategy = strategy;

    public void Sort(List<int> data) => _strategy.Sort(data);
}

// Usage — swap strategy at runtime
var sorter = new Sorter(new QuickSort());
sorter.Sort(new List<int> { 5, 3, 8, 1 });

sorter.SetStrategy(new BubbleSort());   // swap!
sorter.Sort(new List<int> { 5, 3, 8, 1 });
```

## 5️⃣ Observer — Notify Many Listeners 🔄

**Problem:** When something happens, multiple things need to react — without the source knowing about them.

```csharp
// Built-in C# way — events and delegates
public class OrderService
{
    // Event — subscribers attach to this
    public event EventHandler<Order>? OrderPlaced;

    public async Task PlaceOrderAsync(Order order)
    {
        // ... save order ...
        OrderPlaced?.Invoke(this, order);   // notify all listeners ✅
    }
}

// Subscribers — don't know about each other
public class EmailService
{
    public void OnOrderPlaced(object? sender, Order order)
        => Console.WriteLine($"📧 Sending confirmation to {order.CustomerEmail}");
}

public class InventoryService
{
    public void OnOrderPlaced(object? sender, Order order)
        => Console.WriteLine($"📦 Reducing stock for order #{order.Id}");
}

// Wire them up
var orderService = new OrderService();
var emailSvc     = new EmailService();
var inventorySvc = new InventoryService();

orderService.OrderPlaced += emailSvc.OnOrderPlaced;
orderService.OrderPlaced += inventorySvc.OnOrderPlaced;

await orderService.PlaceOrderAsync(new Order { Id = 1, CustomerEmail = "asha@test.com" });
// 📧 Sending confirmation to asha@test.com
// 📦 Reducing stock for order #1
```

## 6️⃣ Decorator — Add Behaviour Without Changing the Class 🔗

**Problem:** Add features (logging, caching, validation) to an existing class without modifying it.

```csharp
public interface IDataService
{
    string GetData(string key);
}

public class DataService : IDataService
{
    public string GetData(string key) => $"Data for {key}";
}

// Logging decorator — wraps the real service
public class LoggingDataService : IDataService
{
    private readonly IDataService _inner;
    public LoggingDataService(IDataService inner) => _inner = inner;

    public string GetData(string key)
    {
        Console.WriteLine($"[LOG] GetData called with key: {key}");
        string result = _inner.GetData(key);
        Console.WriteLine($"[LOG] GetData returned: {result}");
        return result;
    }
}

// Caching decorator — wraps the logging decorator
public class CachingDataService : IDataService
{
    private readonly IDataService _inner;
    private readonly Dictionary<string, string> _cache = new();

    public CachingDataService(IDataService inner) => _inner = inner;

    public string GetData(string key)
    {
        if (_cache.TryGetValue(key, out var cached)) return cached;
        string result = _inner.GetData(key);
        _cache[key] = result;
        return result;
    }
}

// Stack decorators — each adds a layer ✅
IDataService service =
    new CachingDataService(
        new LoggingDataService(
            new DataService()));

service.GetData("user:1");
```

## 📋 Summary Table

| Pattern | Category | Problem it solves |
|---|---|---|
| **Singleton** | Creational | Ensure only one instance exists |
| **Factory** | Creational | Create objects without specifying the exact class |
| **Repository** | Structural | Abstract data access behind an interface |
| **Strategy** | Behavioural | Swap algorithms/behaviours at runtime |
| **Observer** | Behavioural | Notify multiple subscribers of events |
| **Decorator** | Structural | Add behaviour by wrapping, not modifying |

## 🎤 Interview Q&A

**Q1: What is a design pattern?**
A: A reusable, proven solution to a commonly occurring design problem — not code, but a blueprint.

**Q2: What are the three categories of GoF patterns?**
A: Creational (object creation), Structural (composition), Behavioural (communication).

**Q3: What is the Singleton pattern?**
A: Ensures only one instance of a class exists. Use `Lazy<T>` for thread-safe initialization.

**Q4: What is the Factory pattern?**
A: Centralizes object creation logic. The caller asks for an object without knowing the exact class.

**Q5: What is the Repository pattern?**
A: Abstracts data access behind an interface — business logic doesn't know if data comes from SQL, Mongo, or a fake.

**Q6: What is the Strategy pattern?**
A: Defines a family of algorithms behind an interface and makes them swappable at runtime.

**Q7: What is the Observer pattern?**
A: One object (subject) notifies many observers when something happens. C# events are Observer built-in.

**Q8: What is the Decorator pattern?**
A: Wraps an object to add new behaviour without modifying the original class — stackable and composable.

**Q9: How is Strategy different from Factory?**
A: Factory decides *which object to create*. Strategy decides *which algorithm to run*.

**Q10: Which patterns are most common in ASP.NET Core?**
A: Repository (data access), Factory (object creation), Strategy (middleware/services), Decorator (wrapping services), Singleton (registered services).

✅ Patterns are blueprints — not code to copy, but problems to recognize  
✅ **Singleton** — one instance, use `Lazy<T>` for thread safety  
✅ **Factory** — centralize creation, hide concrete types  
✅ **Repository** — abstract data access, makes testing easy  
✅ **Strategy** — swap behaviour at runtime, no `if/else` chains  
✅ **Observer** — C# events are Observer built-in  
✅ **Decorator** — add behaviour by wrapping, not modifying  
✅ Patterns + SOLID + DI = professional-grade architecture
