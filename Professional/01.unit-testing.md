# 🧪 Unit Testing in C#

---

## 🧠 What is Unit Testing?

A **unit test** checks that a small piece of code (a method, a class) does exactly what it should — automatically, every time.

```csharp
// Your code
public int Add(int a, int b) => a + b;

// Your test
[Fact]
public void Add_TwoNumbers_ReturnsSum()
{
    int result = Add(3, 4);
    Assert.Equal(7, result);   // ✅ passes
}
```

> 💡 Tests are your **safety net**. Change code with confidence — if something breaks, a test tells you immediately. No more "hope it works" deployments. 🚀

---

## ⚙️ Setup — xUnit (the standard)

```bash
# Create a test project
dotnet new xunit -o MyApp.Tests
cd MyApp.Tests

# Add reference to the project you're testing
dotnet add reference ../MyApp/MyApp.csproj

# Add FluentAssertions for readable assertions
dotnet add package FluentAssertions

# Add Moq for mocking dependencies
dotnet add package Moq

# Run tests
dotnet test
```

---

## 🏗️ Test Structure — AAA Pattern

Every test follows **Arrange → Act → Assert**:

```csharp
[Fact]
public void Divide_ByNonZero_ReturnsCorrectResult()
{
    // Arrange — set up everything needed
    var calculator = new Calculator();

    // Act — do the thing you're testing
    double result = calculator.Divide(10, 2);

    // Assert — verify the result
    Assert.Equal(5.0, result);
}
```

> ✅ One test = one behaviour. Keep them small and focused.

---

## 🔢 xUnit Attributes

```csharp
// [Fact] — a single test, no parameters
[Fact]
public void IsEven_GivenEvenNumber_ReturnsTrue()
{
    Assert.True(2.IsEven());
}

// [Theory] + [InlineData] — same test, multiple inputs
[Theory]
[InlineData(2,  true)]
[InlineData(3,  false)]
[InlineData(0,  true)]
[InlineData(-4, true)]
public void IsEven_VariousNumbers_ReturnsExpected(int number, bool expected)
{
    Assert.Equal(expected, number.IsEven());
}
```

## ✅ FluentAssertions — Readable Tests

Plain `Assert` is fine. FluentAssertions makes tests read like English:

```csharp
using FluentAssertions;

// Numbers
result.Should().Be(42);
result.Should().BeGreaterThan(10);
result.Should().BeInRange(1, 100);

// Strings
name.Should().Be("Asha");
name.Should().StartWith("A");
name.Should().Contain("sh");
name.Should().NotBeNullOrEmpty();

// Collections
list.Should().HaveCount(3);
list.Should().Contain("C#");
list.Should().BeInAscendingOrder();
list.Should().NotBeEmpty();

// Objects
user.Should().NotBeNull();
user.Should().BeEquivalentTo(expected);   // deep comparison ✅

// Exceptions
Action act = () => calculator.Divide(5, 0);
act.Should().Throw<DivideByZeroException>()
   .WithMessage("*divide by zero*");
```

## 🎭 Moq — Mock Dependencies

Real code has dependencies (databases, APIs, email services). Don't use the real thing in tests — **mock** it:

```csharp
using Moq;

public interface IEmailService
{
    Task SendAsync(string to, string subject, string body);
}

public class UserService
{
    private readonly IEmailService _email;
    public UserService(IEmailService email) => _email = email;

    public async Task RegisterAsync(string email)
    {
        // ... save user ...
        await _email.SendAsync(email, "Welcome!", "Thanks for joining.");
    }
}

// Test — no real emails sent ✅
[Fact]
public async Task Register_ValidEmail_SendsWelcomeEmail()
{
    // Arrange
    var mockEmail = new Mock<IEmailService>();
    var service = new UserService(mockEmail.Object);

    // Act
    await service.RegisterAsync("asha@example.com");

    // Assert — verify SendAsync was called once with the right args
    mockEmail.Verify(e =>
        e.SendAsync("asha@example.com", "Welcome!", It.IsAny<string>()),
        Times.Once);
}
```

## 🔄 Mock Return Values

```csharp
public interface IUserRepository
{
    Task<User?> GetByIdAsync(int id);
}

var mockRepo = new Mock<IUserRepository>();

// Setup: when GetByIdAsync(1) is called, return this user
mockRepo.Setup(r => r.GetByIdAsync(1))
        .ReturnsAsync(new User { Id = 1, Name = "Asha" });

// Setup: return null for any other id
mockRepo.Setup(r => r.GetByIdAsync(It.Is<int>(id => id != 1)))
        .ReturnsAsync((User?)null);
```

## 🌍 Real-World Example: Testing a Service

```csharp
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService _email;

    public OrderService(IOrderRepository repo, IEmailService email)
    {
        _repo = repo;
        _email = email;
    }

    public async Task<bool> PlaceOrderAsync(Order order)
    {
        if (order.Total <= 0) return false;

        await _repo.SaveAsync(order);
        await _email.SendAsync(order.CustomerEmail, "Order Confirmed", $"Order #{order.Id} placed!");
        return true;
    }
}

// Tests
public class OrderServiceTests
{
    private readonly Mock<IOrderRepository> _mockRepo = new();
    private readonly Mock<IEmailService> _mockEmail = new();
    private readonly OrderService _service;

    public OrderServiceTests()
    {
        _service = new OrderService(_mockRepo.Object, _mockEmail.Object);
    }

    [Fact]
    public async Task PlaceOrder_ValidOrder_ReturnsTrueAndSendsEmail()
    {
        // Arrange
        var order = new Order { Id = 1, Total = 500, CustomerEmail = "asha@test.com" };

        // Act
        bool result = await _service.PlaceOrderAsync(order);

        // Assert
        result.Should().BeTrue();
        _mockRepo.Verify(r => r.SaveAsync(order), Times.Once);
        _mockEmail.Verify(e => e.SendAsync("asha@test.com", "Order Confirmed", It.IsAny<string>()), Times.Once);
    }

    [Fact]
    public async Task PlaceOrder_ZeroTotal_ReturnsFalse()
    {
        // Arrange
        var order = new Order { Total = 0 };

        // Act
        bool result = await _service.PlaceOrderAsync(order);

        // Assert
        result.Should().BeFalse();
        _mockRepo.Verify(r => r.SaveAsync(It.IsAny<Order>()), Times.Never);
    }
}
```

## 📋 Summary Table

| Tool | Purpose |
|---|---|
| `xUnit` | Test framework — runs your tests |
| `[Fact]` | Single test, no parameters |
| `[Theory]` + `[InlineData]` | Same test, multiple input sets |
| `FluentAssertions` | Readable assertions (`.Should().Be()`) |
| `Moq` | Mock interfaces — no real dependencies |
| `Mock<T>.Setup()` | Define what the mock returns |
| `Mock<T>.Verify()` | Assert a method was called |
| `It.IsAny<T>()` | Match any argument of type T |
| `dotnet test` | Run all tests in the project |
| AAA Pattern | Arrange → Act → Assert |

## 🎤 Interview Q&A

**Q1: What is unit testing?**
A: Testing individual units of code (methods/classes) in isolation to verify they behave correctly.

**Q2: What is the AAA pattern?**
A: Arrange (set up), Act (run the code), Assert (verify result). Every test should follow this.

**Q3: What is the difference between `[Fact]` and `[Theory]`?**
A: `[Fact]` is a single fixed test. `[Theory]` runs the same test with multiple inputs via `[InlineData]`.

**Q4: Why use mocks?**
A: To isolate the code under test from real dependencies (DB, API, email). Tests stay fast, reliable, and offline.

**Q5: What is Moq?**
A: A library for creating mock objects from interfaces. You control what they return and verify they were called.

**Q6: What does `It.IsAny<T>()` do?**
A: Matches any argument of type T in a Setup or Verify call — when you don't care about the exact value.

**Q7: What is FluentAssertions?**
A: A library that makes assertions read like English — `result.Should().Be(5)` instead of `Assert.Equal(5, result)`.

**Q8: What is test coverage?**
A: The percentage of your code executed by tests. Aim for 70–80%+ on business logic.

**Q9: What's the difference between unit, integration, and end-to-end tests?**
A: Unit = one class in isolation. Integration = multiple classes/DB together. E2E = full app from UI to DB. Unit tests are fastest and most numerous.

**Q10: What makes a good unit test?**
A: Fast, isolated (no real DB/API), repeatable, tests one thing, clear name describing what it tests.

## 🚀 Key Takeaways

✅ Unit tests = your safety net for every code change  
✅ Use **xUnit** — the standard .NET test framework  
✅ Always follow **Arrange → Act → Assert**  
✅ `[Fact]` for single tests · `[Theory]` for data-driven tests  
✅ **FluentAssertions** makes assertions read like English  
✅ **Moq** mocks interfaces — no real DB/API/email in tests  
✅ `Mock.Verify()` — assert a method was actually called  
✅ One test = one behaviour — keep them small and focused  
✅ `dotnet test` runs everything in one command
