# 💉 Dependency Injection in C#

---

## 🧠 What is Dependency Injection?

**Dependency Injection (DI)** means giving a class its dependencies from outside — instead of letting the class create them itself.

```csharp
// ❌ Without DI — class creates its own dependency (tightly coupled)
public class OrderService
{
    private SqlDatabase _db = new SqlDatabase();   // hard-coded! can't swap, can't test
}

// ✅ With DI — dependency is injected from outside (loosely coupled)
public class OrderService
{
    private readonly IDatabase _db;
    public OrderService(IDatabase db) => _db = db;   // injected ✅
}
```

> 💡 DI is like a restaurant. The chef doesn't grow their own vegetables — they're **delivered**. The chef focuses on cooking. Delivery (what arrives) can change without changing the recipe. 🍳

---

## 🔑 Three Types of Injection

```csharp
// 1️⃣ Constructor Injection ✅ (recommended — always use this)
public class OrderService
{
    private readonly IOrderRepository _repo;
    public OrderService(IOrderRepository repo) => _repo = repo;
}

// 2️⃣ Property Injection (use sparingly — optional dependencies)
public class ReportService
{
    public ILogger? Logger { get; set; }   // optional
    public void Generate() { Logger?.LogInformation("Generating..."); }
}

// 3️⃣ Method Injection (rare — dependency only needed in one method)
public class DataProcessor
{
    public void Process(IDataSource source) { /* use source here only */ }
}
```

## 🏗️ The .NET DI Container

.NET has a built-in DI container. You register services once — the container creates and injects them automatically.

```csharp
// Program.cs — register everything here
var builder = WebApplication.CreateBuilder(args);

// Register services
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddScoped<IEmailService, SmtpEmailService>();
builder.Services.AddScoped<OrderService>();
builder.Services.AddSingleton<AppConfig>();
builder.Services.AddTransient<IReportBuilder, PdfReportBuilder>();

var app = builder.Build();
```

---

## ⏳ Service Lifetimes — Critical to Understand

| Lifetime | Created | Destroyed | Use for |
|---|---|---|---|
| **Singleton** | Once, app start | App shutdown | Config, caches, HttpClient |
| **Scoped** | Once per request | End of request | DbContext, services with state |
| **Transient** | Every time requested | After use | Lightweight, stateless services |

```csharp
// Singleton — same instance for entire app lifetime
builder.Services.AddSingleton<IAppConfig, AppConfig>();

// Scoped — one instance per HTTP request (most common for services)
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();

// Transient — new instance every time it's requested
builder.Services.AddTransient<IEmailFormatter, HtmlEmailFormatter>();
```

> 🚨 **Common mistake:** Never inject a Scoped service into a Singleton — the scoped one lives longer than it should. .NET will throw an exception.

## ✅ Constructor Injection in Action

```csharp
// All these get injected automatically ✅
public class OrderService
{
    private readonly IOrderRepository _repo;
    private readonly IEmailService    _email;
    private readonly ILogger<OrderService> _logger;

    public OrderService(
        IOrderRepository repo,
        IEmailService email,
        ILogger<OrderService> logger)
    {
        _repo  = repo;
        _email = email;
        _logger = logger;
    }

    public async Task<bool> PlaceOrderAsync(Order order)
    {
        _logger.LogInformation("Placing order {OrderId}", order.Id);

        await _repo.SaveAsync(order);
        await _email.SendAsync(order.CustomerEmail, "Confirmed!", "Your order is placed.");

        _logger.LogInformation("Order {OrderId} placed successfully", order.Id);
        return true;
    }
}
```

## 🧪 DI Makes Testing Easy

Because you inject interfaces, you swap real implementations with fakes in tests — no container needed:

```csharp
// In tests — inject fakes directly ✅
[Fact]
public async Task PlaceOrder_ValidOrder_SavesAndSendsEmail()
{
    // Arrange
    var mockRepo  = new Mock<IOrderRepository>();
    var mockEmail = new Mock<IEmailService>();
    var mockLog   = new Mock<ILogger<OrderService>>();

    var service = new OrderService(mockRepo.Object, mockEmail.Object, mockLog.Object);
    var order   = new Order { Id = 1, CustomerEmail = "asha@test.com" };

    // Act
    bool result = await service.PlaceOrderAsync(order);

    // Assert
    result.Should().BeTrue();
    mockRepo.Verify(r => r.SaveAsync(order), Times.Once);
}
```

## 🔄 Registering Open Generics

```csharp
// Register IRepository<T> for any T at once
builder.Services.AddScoped(typeof(IRepository<>), typeof(SqlRepository<>));

// Now this works for any entity automatically
public class ProductService
{
    public ProductService(IRepository<Product> repo) { }
}

public class UserService
{
    public UserService(IRepository<User> repo) { }
}
```

## 🌍 Real-World: Full Registration Example

```csharp
// Program.cs
var builder = WebApplication.CreateBuilder(args);

// Framework services
builder.Services.AddControllers();
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// App services
builder.Services.AddScoped<IProductRepository, SqlProductRepository>();
builder.Services.AddScoped<IOrderRepository,   SqlOrderRepository>();
builder.Services.AddScoped<IProductService,    ProductService>();
builder.Services.AddScoped<IOrderService,      OrderService>();

// Singletons
builder.Services.AddSingleton<IAppConfig, AppConfig>();

// HTTP client (registered as singleton internally)
builder.Services.AddHttpClient<IWeatherService, WeatherService>();

var app = builder.Build();
app.MapControllers();
app.Run();

// Controller — dependencies auto-injected ✅
[ApiController]
[Route("api/orders")]
public class OrdersController : ControllerBase
{
    private readonly IOrderService _orderService;

    public OrdersController(IOrderService orderService)
        => _orderService = orderService;

    [HttpPost]
    public async Task<IActionResult> PlaceOrder(Order order)
    {
        bool success = await _orderService.PlaceOrderAsync(order);
        return success ? Ok() : BadRequest();
    }
}
```

## 📋 Summary Table

| Concept | Code | Notes |
|---|---|---|
| Register Singleton | `AddSingleton<I, T>()` | One instance, whole app |
| Register Scoped | `AddScoped<I, T>()` | One per request |
| Register Transient | `AddTransient<I, T>()` | New every time |
| Constructor inject | `public Service(IDep dep)` | Preferred approach ✅ |
| Open generic | `AddScoped(typeof(I<>), typeof(T<>))` | Register for all T at once |
| Manual resolve | `provider.GetService<T>()` | Avoid — use injection instead |

## 🎤 Interview Q&A

**Q1: What is Dependency Injection?**
A: Providing a class its dependencies from outside rather than having it create them — promotes loose coupling and testability.

**Q2: What are the three lifetimes in .NET DI?**
A: Singleton (one for whole app), Scoped (one per request), Transient (new every time).

**Q3: Which injection type is recommended?**
A: Constructor injection — dependencies are explicit, required, and visible.

**Q4: What is the DI container?**
A: A framework component that stores registrations and creates/injects objects automatically.

**Q5: Why is DI important for unit testing?**
A: You can inject mocks instead of real implementations — no database or network needed in tests.

**Q6: What happens if you inject a Scoped service into a Singleton?**
A: .NET throws an `InvalidOperationException` at startup — the scoped service would outlive its intended lifetime.

**Q7: What is the difference between `AddScoped` and `AddTransient`?**
A: Scoped = one instance per HTTP request (shared within that request). Transient = new instance every single time it's requested.

**Q8: How do you register a generic repository?**
A: Use open generics:
```csharp
services.AddScoped(typeof(IRepository<>), typeof(SqlRepository<>));
```

**Q9: What is the Service Locator anti-pattern?**
A: Calling `provider.GetService<T>()` directly inside classes — avoids this. Use constructor injection instead. Makes dependencies hidden and hard to test.

**Q10: What is `IServiceCollection`?**
A: The interface you call `Add*` methods on to register services. `builder.Services` in ASP.NET Core is an `IServiceCollection`.

## 🚀 Key Takeaways

✅ DI = give dependencies from outside, don't create them inside  
✅ Use **constructor injection** — always the preferred approach  
✅ Three lifetimes: **Singleton** · **Scoped** · **Transient**  
✅ **Scoped** is the most common for services in web apps  
✅ Never inject Scoped into Singleton — runtime exception  
✅ DI + interfaces = easily **mockable** and **testable** code  
✅ Register in `Program.cs`, use everywhere via constructor  
✅ Open generics let you register `IRepo<T>` for any T at once
